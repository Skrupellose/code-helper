# Code Helper Plan Workbench

## 目标

当需要为迁移、升级、重构、多阶段功能建设或长期协作任务生成计划文档时，输出不能只是“可阅读”的说明文档，而必须是能持续推进、记录阻塞、支持回归、恢复上下文的执行工作台。

## 适用场景

- 旧项目迁移到新框架
- 大型功能分阶段建设
- 技术债清理与结构重构
- 页面、组件、接口、状态层存在明显依赖关系的任务
- 需要长期推进、多人协作或中途暂停再恢复的任务
- 用户要求生成计划文档、阶段计划、执行工作台、状态跟踪或计划优化

## 最终产物

默认生成或维护以下文件：

- code-helper-docs/plan-doc/<中文功能名>.md：执行计划工作台
- code-helper-docs/result-doc/<中文功能名>/实施记录.md：阶段或小节点实施记录
- code-helper-docs/status-doc/<中文功能名>-状态.md：当前状态驾驶舱
- code-helper-docs/result-doc/<中文功能名>/手工测试.md：页面手工测试文档

所有最终产物必须使用中文命名，并在文档内使用中文总结当前目标、完成情况、验证结论、风险和下一步。

最终计划必须同时具备：

1. 总纲
2. 分层顺序
3. 组件计划
4. 页面计划
5. 验收标准
6. 状态跟踪
7. 阻塞回归入口
8. 当前推进建议

## 标准生成顺序

### Step 1：确认目标与约束

先明确：

- 项目最终目标
- 当前最重要的优先级
- 当前阶段边界
- 必须写进计划的验收标准
- 用户补充约束
- 当前仓库现状和已有入口规则

输出主原则、总体约束和阶段边界。

### Step 2：先出总纲计划

先只输出：

- P0 / P1
- 总阶段顺序
- 每阶段目标
- 总体验收口径

不要一开始直接写细表。

### Step 3：按依赖关系重排顺序

必须按依赖闭环重排，而不是按页面直觉排序。

推荐主顺序：

1. 基础目录
2. 类型定义
3. API / 请求层
4. 状态与守卫
5. 应用骨架
6. i18n 最小闭环
7. 通用组件与布局层
8. 组件组迁移或建设
9. 页面迁移或建设
10. 回归与横向升级

输出真实执行顺序和每层依赖说明。

### Step 4：做目录或模块级策略

先分析当前项目结构，输出：

- 可整包前置迁移或前置建设的目录 / 模块
- 需要拆分推进的目录 / 模块
- 强依赖后置目录 / 模块
- 建议暂时跳过项

输出目录优先级策略、迁移或建设顺序、暂时跳过项。

### Step 5：生成基础阶段计划

基础阶段至少覆盖：

- 前置基础目录
- 类型定义与数据模型
- API 与请求方法
- 状态管理与路由守卫
- 最小运行骨架
- i18n 最小闭环
- 通用组件与布局层

通用组件与布局层不能只写目标，必须也有执行表。

### Step 6：页面前先做组件计划

页面不是第一迁移单元，组件才是。进入页面计划前，必须先生成：

1. 页面到组件依赖总表
2. 组件分组
3. 组件优先级：P0 / P1
4. 单组件 checklist
5. 组件组真实执行顺序
6. 组件组总览表和逐组件组执行表

### Step 7：再做页面逐页执行表

只有组件计划完成后，才进入页面计划。

每页必须包含：

- 前置依赖
- 执行任务
- UI 验收
- 功能验收
- 完成定义

页面阶段必须先有页面总览状态表，再有逐页执行表。

### Step 8：改造成执行工作台

最终文档必须统一补上：

- 基础阶段总览 + 明细
- 组件阶段总览 + 明细
- 页面阶段总览 + 明细
- 当前推进建议
- 状态枚举规范
- 阻塞和回归入口

## 表格规范

所有总览表和明细表统一使用 4 列：

| 子任务 | 已完成 | 未完成 | 备注 |
|---|---|---|---|
| 示例 | 已完成内容 | 剩余内容 | 状态：未开始；阻塞原因 / 依赖说明 / 回归入口 |

备注第一句必须写：

- 状态：未开始
- 状态：进行中
- 状态：部分完成
- 状态：被阻塞
- 状态：已完成

备注还应说明：

- 是否被阻塞
- 阻塞点是什么
- 依赖哪个后续流程
- 后续从哪里回归
- 是否允许先跳过继续推进

## 当前推进建议

文档前部必须包含“当前推进建议”，明确：

1. 现在先做什么
2. 下一步接什么
3. 第一个组件组何时开始
4. 第一个页面何时开始
5. 当前默认执行主线是什么

这样每次打开文档，第一眼就知道下一步。

## 页面与测试规则

- 页面计划必须拆成 UI 验收和功能验收，不能只写“完成页面开发”。
- 页面、组件交互、视觉和真实浏览器链路只生成严格手工测试文档。
- 工具自身只执行纯逻辑测试，例如函数单元测试、数据转换测试、非浏览器集成测试。
- 手工测试文档必须包含测试环境、前置数据、操作步骤、预期结果、回归范围和阻塞记录。

## 状态文档分工

- plan-doc：计划工作台，记录阶段、依赖、任务表和验收标准。
- result-doc：实施记录，使用中文总结实际改动、验证结论、临时失败、风险和后续。
- status-doc：当前驾驶舱，只保留当前状态、下一步、阶段进度、关键结论、关键索引、仍影响判断的风险点、最近一次更新。

状态文档不是流水账，不写完整命令输出、大段实现过程或已解决的旧风险。

## 成型标准

文档满足以下条件才算最终版：

- 有明确总顺序
- 有基础层、组件层、页面层
- 有 UI 验收与功能验收
- 有总览表和明细表
- 有状态规范
- 有阻塞回归机制
- 有当前推进建议
- 能直接拿来推进，而不是还要二次设计

## 边界规则

- 不要创建英文任务文档名，例如 implementation.md、manual-test.md 或 <feature>-status.md；新文档统一使用 实施记录.md、手工测试.md 和 <中文功能名>-状态.md。
- 不要跳过总纲直接写细表。
- 不要跳过组件计划直接进入页面计划。
- 不要按页面直觉排序，必须按依赖闭环排序。
- 不要把页面测试写成默认自动化测试任务。
- 不要把 status-doc 写成历史流水账。
- 不要把所有细节塞进入口规则或状态驾驶舱，细节放 result-doc。
