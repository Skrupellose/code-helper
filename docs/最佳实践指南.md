# code-helper 最佳实践指南

这份文档面向第一次接触 `code-helper` 的用户。它不只列命令，而是按真实使用顺序说明：怎么安装、怎么初始化项目、一级功能分别解决什么问题，以及配置完成后如何让 Codex、Claude Code 或 GitHub Copilot 通过项目级 Skills 稳定参与开发。

## 一句话理解 code-helper

`code-helper` 是一个给 AI 编码协作项目用的 CLI。它做三件事：

1. 把项目规则、任务计划、实施记录、状态记录和归档目录搭起来。
2. 给 agent 提供可读取的项目级 Skills，让它知道什么时候该拆计划、写手工测试、做完成检查、询问是否更新长期记忆。
3. 在任务结束前做兜底检查，避免“代码改完了，但计划、验收、风险和归档没人处理”。

它不是自动开发平台，也不是测试框架。`plan` 和 `manual-test` 命令只生成结构化模板，真正的需求拆解、上下文阅读、测试步骤补全和风险判断，仍然由 agent 结合代码和文档完成。

## 安装方式怎么选

### 方式一：npx 临时使用

如果你只是想在一个项目里快速初始化，或者不确定是否要长期使用，可以直接运行：

```bash
npx @skrupellose/code-helper@latest
```

这会打开交互菜单。也可以直接执行某个命令：

```bash
npx @skrupellose/code-helper@latest init
npx @skrupellose/code-helper@latest check
```

临时使用的好处是不用改 `package.json`，也不用关心项目里装了哪个版本。适合试用、一次性整理项目，或在还没决定协作流程前先体验。

它的不足是团队成员每次运行时都依赖 npm latest，版本可能不完全一致。Node 项目长期使用时，可以改为项目内安装；非 Node 项目可以在团队说明里固定推荐的 code-helper 版本或统一工具链入口。

### 方式二：安装到项目开发依赖

Node 项目长期使用时，推荐安装为开发依赖：

```bash
npm i -D @skrupellose/code-helper
npx code-helper init
```

之后日常命令使用：

```bash
npx code-helper <命令>
```

推荐 Node 项目内安装的原因很简单：团队和 CI 看到的是同一个版本，README、入口文档、Skills 和 hooks 刷新时也更可控。需要升级时，先升级 npm 包，再刷新本地受控资产：

```bash
npm i -D @skrupellose/code-helper@latest
npx code-helper update
```

如果希望把常用命令写进 `package.json`，可以执行：

```bash
npx code-helper npm-scripts install
```

它只会写入缺失的脚本，不覆盖已有同名脚本。

### Go、Java 等非 Node 项目怎么用

code-helper 本身是 npm 分发的 CLI，所以运行机器需要有 Node 和 `npx`。这个要求只发生在工具运行层，不等于你的业务项目必须是 Node 项目。

它管理的是项目内协作文档、入口文件、项目级 Skills 和可选 Hooks，例如 `code-helper-docs/`、`AGENTS.md`、`CLAUDE.md`、`.github/copilot-instructions.md`、`.agents/skills/`、`.claude/skills/` 和 `.github/skills/`。这些内容和 Go、Java、Python、Rust 或其他语言的源码结构没有绑定关系。

非 Node 项目建议这样选：

- 不想引入 `package.json`：优先用 `npx @skrupellose/code-helper@latest` 打开菜单，或用 `npx @skrupellose/code-helper@latest <命令>` 精确执行。
- 团队有统一工具链镜像：可以全局安装或用内部脚本包装，但仍然建议在说明里写清使用的 code-helper 版本。
- Go、Java 项目不需要执行 `npm i -D @skrupellose/code-helper`，也不需要执行 `npm-scripts install`。

语言无关的能力包括：`init`、`update`、`plan`、`manual-test`、`finish`、`tasks`、`archive`、`skills register` 和 `check`。它们主要读写协作文档和 agent 配置，不要求项目里有 `package.json`。

和 Node/npm 绑定更强的能力包括：项目内 `npm i -D`、依赖项目本地包解析的 `npx code-helper <命令>`、`npm-scripts install`、菜单里的 npm 版本升级快捷项。Git hook 和 Agent hook 运行时也需要环境里能调用 Node、`npx`，或能调用项目内可用的 `code-helper`。

## 第一次接入项目

### 1. 先用菜单走一遍初始化

第一次接入时，如果你还不熟悉参数，建议先打开交互菜单：

```bash
npx code-helper
```

如果项目里还没有安装，也可以临时运行：

```bash
npx @skrupellose/code-helper@latest
```

进入菜单后，选择“项目准备”里的“初始化/刷新项目配置”。在支持方向键的终端里，可以用方向键移动，用空格或回车确认；如果当前终端不支持方向键交互，code-helper 会自动退回数字菜单，这时输入 `1` 选择“初始化/刷新项目配置”即可。

这个菜单项适合第一次接入，也适合你只是想刷新入口文档、规则模板、Skills 或 hooks，但不想记命令参数的时候。它会按当前项目状态执行初始化：已有入口文件的项目会尽量沿用现有目标；完全没有入口文件的新项目，会继续询问你要接入哪些 agent 工具。

### 2. 新项目如何选择 agent 目标

如果项目里还没有 `AGENTS.md`、`CLAUDE.md` 或 GitHub Copilot 入口文件，交互式 init 会让你选择 agent 工具。方向键菜单里默认勾选 Codex，你可以直接回车确认，也可以用空格调整选择；数字菜单里可以输入 `1` 选择 Codex，输入 `2` 选择 Claude Code，输入 `3` 选择 GitHub Copilot，输入 `A` 选择全部。

目标选择可以按下面理解：

- 你现在主要用 Codex，或者只是想先把流程跑起来：选默认的 Codex。
- 团队明确只用 Claude Code 或 GitHub Copilot：选对应目标，不需要同时创建其他入口。
- 同一个仓库明确要同时支持 Codex、Claude Code 和 GitHub Copilot：再选全部。

不建议新项目一上来就为了“以后可能会用”选择全部。`all` 会创建三类 agent 的入口和项目级 Skills，适合明确多工具协作的仓库；普通单 agent 项目选实际使用的目标更清爽。

如果项目已经存在入口文件，init 会根据当前项目推断目标，通常不会再问你：

| 已存在内容                                                 | 默认目标           |
| ----------------------------------------------------- | -------------- |
| `AGENTS.md`                                           | Codex          |
| `CLAUDE.md`                                           | Claude Code    |
| `.github/copilot-instructions.md` 或 `.github/skills/` | GitHub Copilot |

### 3. 命令方式怎么写

菜单方式适合第一次接入、演示流程、临时维护，或者还不确定该传哪些参数的时候。命令方式适合写进 README、脚本、CI，或者你已经知道目标，需要让别人精确复现时使用。

等你明确目标后，可以直接运行：

```bash
npx code-helper init
```

`init` 会创建或更新这些内容：

- `.code-helper/`：工具配置、内置模板和 hook 包装脚本。
- `code-helper-docs/user-rules/`：长期协作规则。
- `code-helper-docs/plan-doc/`：任务计划文档。
- `code-helper-docs/result-doc/`：实施记录和手工测试文档。
- `code-helper-docs/status-doc/`：当前状态记录。
- `AGENTS.md`、`CLAUDE.md`、`.github/copilot-instructions.md`：对应 agent 的入口文档。
- `.agents/skills/`、`.claude/skills/`、`.github/skills/`：对应 agent 的项目级 Skills。

它不会整份覆盖你的入口文档，只维护 `<!-- code-helper:start -->` 和 `<!-- code-helper:end -->` 之间的受控区块。你自己写的项目规则应放在受控区块外，长期规则放到 `code-helper-docs/user-rules/`。

你也可以显式指定：

```bash
npx code-helper init codex
npx code-helper init claudecode
npx code-helper init githubcopilot
npx code-helper init all
```

最佳实践是：项目实际用哪个 agent，就注册哪个目标。不要为了“以后可能会用”一上来就 `all`，否则项目里会多出不必要的入口和 Skills 目录。

非交互环境不会默认安装全部目标，因为这会在项目里写入你可能根本不用的 agent 配置。脚本和 CI 里如果确实要注册目标，应显式传 `codex`、`claudecode`、`githubcopilot` 或 `all`。

### 4. 后续升级和刷新怎么理解

菜单里的“初始化/刷新项目配置”主要用于创建或更新当前项目里的 code-helper 入口、规则和项目级能力；它不是 npm 包升级器。

如果你无参数运行 `npx code-helper` 时检测到 npm 上有新版，菜单顶部可能出现“更新到最新版本”的快捷项。这个快捷项和普通“项目准备”分组是分开的：它会先升级当前项目安装的 code-helper npm 包，再启动新版 `code-helper update` 刷新当前项目已经启用的入口、Skills 和 Hooks。

命令方式下，升级后刷新本项目用：

```bash
npx code-helper update
```

`update` 只刷新当前项目已经启用、已经注册或能从现有入口推断出来的 code-helper 资产。它不会自动开启未启用能力，也不会凭空创建未使用 agent 的入口文件。

日常理解可以很简单：

- 第一次接入：先运行 `npx code-helper`，从菜单选“项目准备 / 初始化/刷新项目配置”。
- 需要写进 README、脚本或 CI：再补充 `npx code-helper init`。
- code-helper 升级后刷新本项目：用 `update`，或使用菜单顶部的“更新到最新版本”快捷项。
- 本仓库开发者同步模板：才用 `sync-local`，普通用户不需要。

## 一级功能怎么用

无参数运行 `npx code-helper` 会打开交互菜单。主菜单按用途分为“项目准备 / 任务推进 / 项目维护 / 工具设置”。

“第一次接入项目”章节已经说明了菜单的基本走法。后面每个一级功能都会同时保留命令示例和菜单入口，两种方式可以混用：菜单适合探索和临时操作，命令适合沉淀到 README、脚本、CI 或本地 alias。

### 初始化/刷新项目配置

在菜单里选择“项目准备”下的“初始化/刷新项目配置”。如果项目里已经安装了旧版本并且检测到 npm 上有新版，菜单顶部还可能出现“更新到最新版本”的快捷项，用来先升级包再刷新当前项目入口、Skills 和 Hooks。

这个入口适合第一次接入、升级 code-helper 后刷新项目、补齐入口文档、刷新规则模板或同步已注册 Skills。它存在的原因是：agent 协作依赖项目内的入口文档、专题规则、Skills 和过程目录。如果这些资产散落、版本不一致，agent 很容易漏读规则，或者把短期任务状态写进长期记忆。

如果要把初始化或刷新写进 README、脚本或 CI，对应命令是：

```bash
npx code-helper init
npx code-helper update
```

### 生成任务计划模板

在菜单里选择“任务推进”下的“生成任务计划模板”。菜单会提示你输入或拖拽需求文档路径，再输入中文功能名称；功能名称可以留空，工具会尽量从需求标题或中文文件名推断。

它适合任务比较完整、需要分阶段推进、可能跨多次会话或多人协作的场景。菜单完成后会生成：

- `code-helper-docs/plan-doc/订单管理升级.md`
- `code-helper-docs/result-doc/订单管理升级/实施记录.md`
- `code-helper-docs/status-doc/订单管理升级-状态.md`

注意：CLI 只生成模板和需求摘要，不代表已经完成完整计划。生成后可以直接对 agent 说：“根据 `docs/订单管理需求.md` 补全开发计划”“把订单管理升级拆成可执行阶段”“按依赖顺序整理执行计划”。

如果要在文档或脚本里复现同一步，对应命令是：

```bash
npx code-helper plan docs/订单管理需求.md 订单管理升级
```

### 生成手工测试模板

在菜单里选择“任务推进”下的“生成手工测试模板”。如果项目里已有活动任务，菜单会优先让你从任务列表里选；没有合适任务时，也可以手动输入功能名称。

它适合页面、可视化、浏览器真实链路、人工业务验收或回归测试场景。code-helper 的默认策略是：页面和真实浏览器链路由用户手工验收，工具只执行纯逻辑测试。

菜单完成后会生成：

```text
code-helper-docs/result-doc/订单管理升级/手工测试.md
```

CLI 只创建结构化模板。完整测试环境、前置数据、操作步骤、预期结果、回归范围和阻塞记录，可以对 agent 说：“补一份人工验收清单”“把这个页面改动整理成手工测试步骤”“补全回归测试范围”。

如果要写进 README 或脚本，对应命令是：

```bash
npx code-helper manual-test 订单管理升级
```

### 检查功能完成情况

在菜单里选择“任务推进”下的“检查功能完成情况”。菜单会优先列出活动任务，适合你不确定当前功能名、只想从现有任务里挑一个做收尾检查的场景。

它适合在完成一个小节点、准备最终回复、准备切换任务、准备归档前使用。`finish` 只读取并判断，不会自动更新长期记忆、不会自动归档、不会提交代码。它会检查任务文档是否齐全、状态记录是否有当前执行节点、计划里是否还有未开始或进行中的项，并提示是否需要询问用户更新长期记忆或归档。

它存在的原因是：很多协作问题不是代码没改完，而是“改完后没有同步计划、没有记录验证、没有说明风险、没有问是否归档”。`finish` 把这些收尾动作显式化。

如果要在最终回复前固定执行或写进脚本，对应命令是：

```bash
npx code-helper finish 订单管理升级
npx code-helper finish 订单管理升级 --check-only
```

### 查看任务列表

在菜单里选择“项目维护”下的“查看任务列表”。这是恢复上下文时最直接的入口：先看哪些任务还在活动区，哪些已经归档，哪些出现了 mixed 冲突。

它会列出任务的状态：

- `active`：仍在顶层 `plan-doc`、`result-doc`、`status-doc` 中。
- `archived`：已经进入对应 `archive/` 目录。
- `mixed`：顶层和归档目录里同时存在同名任务，需要人工确认。

适合恢复上下文、确认当前有哪些活动任务、归档后复查状态时使用。如果要写进脚本或需要 JSON 输出，对应命令是：

```bash
npx code-helper tasks
npx code-helper tasks --json
```

### 归档已完成任务

在菜单里选择“项目维护”下的“归档已完成任务”。菜单会优先从活动任务中选择要归档的功能，避免你手动输入错功能名；如果出现 mixed 冲突，仍然需要人工确认后再处理。

它会把任务相关文档移动到：

- `code-helper-docs/plan-doc/archive/`
- `code-helper-docs/result-doc/archive/`
- `code-helper-docs/status-doc/archive/`

归档不是删除，而是把已结束任务移出活动区。这样后续 agent 读取当前任务时，不会把已经结束的计划误当作还要推进的工作。

如果要写进交接说明或由用户确认后精确执行，对应命令是：

```bash
npx code-helper archive 订单管理升级
```

如果出现 `mixed` 冲突，默认会停止并提示人工确认。只有你确认保留归档版本、清理活动副本时，才使用：

```bash
npx code-helper archive 订单管理升级 --resolve-mixed
```

### 检查协作规范

在菜单里选择“项目维护”下的“检查协作规范”。如果只是阶段结束前做一次结构检查，用菜单点选更省心；如果要在提交前或脚本里兜底，保留命令方式更合适。

默认只输出检查结果，不写报告文件。它会检查入口文档、规则目录、计划/结果/状态目录、中文任务文档、测试策略和归档结构。适合阶段结束、提交前、本地规则整理后运行。

如果需要持久化报告，或者要把检查写进脚本，对应命令是：

```bash
npx code-helper check
npx code-helper check --write-report
```

### 功能管理

在菜单里选择“工具设置”下的“功能管理”。这是交互菜单里的综合入口，用来应用或取消项目级 Skills、Agent hooks 和 Git hook，更适合不想记命令参数的用户。

它和底层 `features enable/disable` 不完全一样。`features` 是配置开关，偏脚本接口；“功能管理”会直接执行注册、安装或取消动作，并同步内部配置状态。

常见选择：

- 应用项目级 Skills：让 agent 在当前项目能发现 code-helper Skills。
- 取消项目级 Skills：只移除 code-helper 管理的 skills，不碰用户自己的 skills。
- 应用 Agent hooks：给 Codex 或 Claude Code 安装完成检查提醒。
- 应用 Git hook：安装提交前的 `code-helper check` 兜底。

### 管理项目 Skills

在菜单里选择“工具设置”下的“管理项目 Skills”。子菜单里可以查看注册状态、按当前项目注册或取消注册，也可以只注册 Codex、Claude Code、GitHub Copilot，或注册全部目标。

`skills register` 不带目标时，会按当前项目已有入口文件自动选择目标。无法识别时会跳过，并提示你显式传 `codex`、`claudecode`、`githubcopilot` 或 `all`。

`skills register all` 是强制注册全部三类目标，适合一个仓库同时明确支持 Codex、Claude Code 和 GitHub Copilot 的情况。普通单 agent 项目不建议默认使用。

`skills doctor` 用来检查项目级 Skills 结构和质量，`skills audit` 会根据项目状态给出建议。

如果要写进 README、脚本或排查记录，对应命令是：

```bash
npx code-helper skills list
npx code-helper skills register
npx code-helper skills register codex
npx code-helper skills register claudecode
npx code-helper skills register githubcopilot
npx code-helper skills register all
npx code-helper skills unregister codex
npx code-helper skills doctor
npx code-helper skills audit
```

### 管理 Hooks

在菜单里选择“工具设置”下的“管理 Hooks”。子菜单会把 Git pre-commit hook、Codex Agent hook 和 Claude Code Agent hook 分开列出，适合避免把提交前检查和 agent 收尾提醒混在一起。

`hooks install` 必须显式传目标，不会把空目标解释成全部。这是为了避免误装 Git hook 或误改 agent 配置。

如果要写进团队说明或脚本，对应命令是：

```bash
npx code-helper hooks list
npx code-helper hooks install codex
npx code-helper hooks install claudecode
npx code-helper hooks install agent
npx code-helper hooks install git
npx code-helper hooks uninstall codex
```

支持目标：

| 目标           | 含义                                   |
| ------------ | ------------------------------------ |
| `codex`      | 安装 Codex Agent Stop hook             |
| `claudecode` | 安装 Claude Code Agent Stop hook       |
| `agent`      | 安装 Codex 和 Claude Code 的 Agent hooks |
| `git`        | 安装 Git pre-commit hook               |
| `all`        | 安装 Git、Codex、Claude Code 三类 hook     |

GitHub Copilot 当前不支持 code-helper 的 Agent hook，因此不会出现在 Agent hook 安装目标里。

## Skills 注册目标有什么差异

code-helper 的内置 Skills 会注册到不同 agent 能识别的项目级目录：

| 目标             | 写入目录                           | 入口文档                              |
| -------------- | ------------------------------ | --------------------------------- |
| Codex          | `.agents/skills/code-helper-*` | `AGENTS.md`                       |
| Claude Code    | `.claude/skills/code-helper-*` | `CLAUDE.md`                       |
| GitHub Copilot | `.github/skills/code-helper-*` | `.github/copilot-instructions.md` |

这三个目录都只影响当前项目，不是用户全局 Skills。

推荐选择：

- 只用 Codex：`npx code-helper skills register codex`
- 只用 Claude Code：`npx code-helper skills register claudecode`
- 只用 GitHub Copilot：`npx code-helper skills register githubcopilot`
- 一个仓库明确三者都要支持：`npx code-helper skills register all`
- 项目已经有入口文件，想按当前项目状态注册：`npx code-helper skills register`

不要把 `.code-helper/skills/` 当成 agent 自动识别目录。它是 code-helper 的内置模板源；真正让 agent 发现 Skills 的动作，是 `skills register` 写入对应 agent 的项目级 skills 目录。

## 配置完成后的 Skills 开发工作流

完成 `init` 和 `skills register` 后，推荐按下面的节奏推进开发。

### 1. 开始任务：把目标说清楚

你可以这样对 agent 提需求：

> 新增登录页，请先读取 AGENTS.md 和现有路由结构，按 code-helper 流程生成计划文档。页面验收只生成手工测试文档，不要跑浏览器自动化。

这样写有三个好处：

1. 明确任务目标。
2. 提醒 agent 读取入口文档和规则。
3. 明确测试边界，避免它误跑不需要的页面自动化。

如果任务已经有需求文档，先把目标说成可执行的请求：

> 根据 `docs/登录页需求.md` 补全开发计划，把这个需求拆成可执行阶段，并按依赖顺序给出下一步。

如果还没有任务模板，通过菜单选择“任务推进 / 生成任务计划模板”，拖入 `docs/登录页需求.md`，功能名填写“新增登录页”。如果要把这一步写进 README 或脚本，对应命令是：

```bash
npx code-helper plan docs/登录页需求.md 新增登录页
```

如果团队已经熟悉内部名称，也可以补一句：“这一步按 `code-helper-plan-workbench` 的要求处理。”

### 2. 生成计划：先总纲，再拆依赖

计划不是任务清单越多越好，而是要让后续执行知道顺序。好的计划应包含：

- 阶段边界。
- 基础能力、核心实现、集成验收、发布整理。
- 每项的完成定义。
- 当前执行节点和下一步建议。
- 自动验证与手工验收边界。

你可以直接说：“把这个需求变成可执行计划”“生成迁移计划”“按依赖顺序拆任务”“先总纲后细化”。这类表达会把重点放在计划质量上，而不是让用户记内部 skill 名称。`code-helper-plan-workbench` 的价值在于它会要求 agent 按依赖链路拆解，而不是按需求文档顺序或界面直觉直接开做。

### 3. 执行开发：一次只推进当前节点

开发过程中，agent 应先读：

1. `code-helper-docs/status-doc/<中文功能名>-状态.md`
2. `code-helper-docs/plan-doc/<中文功能名>.md`
3. `code-helper-docs/result-doc/<中文功能名>/实施记录.md`

执行时只推进当前节点。完成后把实际改动、验证结论、风险和下一步写入实施记录，并更新状态记录里的当前执行节点。

这样做的原因是：多轮会话或多人协作时，后来的 agent 不需要从聊天记录里猜进度，直接读 status-doc 就能知道现在该接哪一步。

### 4. 补充手工测试：页面和真实链路交给用户验收

涉及页面、可视化、组件交互、真实浏览器链路或人工业务验收时，可以先对 agent 说：“补一份人工验收清单”“把这个页面改动整理成手工测试步骤”“补全回归测试范围”。如果还没有测试文档，再通过菜单选择“任务推进 / 生成手工测试模板”。

如果要用命令生成模板，对应命令是：

```bash
npx code-helper manual-test 新增登录页
```

模板生成后，再让 agent 补全：

- 测试环境。
- 前置账号和数据。
- 操作步骤。
- 可判定的预期结果。
- 回归范围。
- 阻塞记录。

如果团队已经熟悉内部名称，也可以说：“请按 `code-helper-manual-test-workbench` 补全这份手工测试。”

如果只是纯函数、数据转换、CLI 输出或非浏览器集成逻辑，则可以让工具执行自动化测试，并把命令和结论写入实施记录。

### 5. 完成检查：先判断，再总结

准备最终回复或切换任务前，先让 agent 做一次收尾判断，例如：“完成前做一次收尾检查”“检查这个小节点是否真的完成”“切换下一个任务前先看还有没有未完成项”。

如果要用命令辅助检查，对应命令是：

```bash
npx code-helper finish 新增登录页 --check-only
```

如果输出还有未开始、进行中、部分完成或被阻塞的节点，应继续当前任务，而不是直接归档或开新任务。

如果输出提示需要询问用户更新长期记忆、归档或选择下一个任务，最终回复里必须明确提出，不要自动执行。

### 6. 归档：用户确认后再执行

功能整体完成、过程文档和验证结论都更新后，再询问用户是否归档。用户确认后，优先从菜单选择“项目维护 / 归档已完成任务”。如果要写进交接命令，对应命令是：

```bash
npx code-helper archive 新增登录页
npx code-helper tasks
```

归档后，活动区只留下还要继续推进的任务。

### 7. 是否更新长期记忆：只沉淀稳定规则

不是每次改代码都要更新长期记忆。适合写入长期规则的内容包括：

- 新的协作边界。
- 稳定的测试策略。
- 项目结构约定。
- 发布流程约束。
- 长期有效的工具使用偏好。

不适合写入长期记忆的内容包括：

- 一次性调试过程。
- 临时失败细节。
- 某个功能的短期状态。
- 可以直接从代码看出来的普通实现细节。

当需要更新时，先对 agent 说：“这个规则以后都这样，先问我是否沉淀到长期记忆”“把这次形成的稳定约束整理成记忆更新草案”。用户确认后，再定向更新 `code-helper-docs/user-rules/` 下的专题规则。熟悉内部名称的团队，也可以补充说按 `code-helper-memory-tuning` 处理。

## 用户怎么说，更容易触发正确 Skills

Skills 通常根据任务描述触发。你不需要记住 `code-helper-plan-workbench` 这类内部名称，先把想做的事说清楚，agent 会根据关键词和上下文选择合适流程。内部名称可以作为补充：团队已经熟悉这些名称时，再显式说 `code-helper-plan-workbench`、`code-helper-manual-test-workbench` 等，用来减少歧义。

推荐说法：

- “根据 `docs/需求.md` 补全开发计划，再开始实现。”
- “把这个需求拆成实施阶段，按依赖顺序整理执行计划。”
- “生成迁移计划，先给总纲，再拆基础能力、核心实现和集成验收。”
- “这是一个页面改动，页面验收只生成手工测试文档。”
- “补一份人工验收清单，覆盖操作步骤、预期结果和回归范围。”
- “完成这个小节点后，请做一次收尾检查并更新实施记录。”
- “如果这次形成稳定规则，先问我是否沉淀到长期记忆。”
- “这个任务完成后，先问我是否归档，不要自动归档。”

常见关键词可以这样理解：

| 你可以怎么说                                                                       | 对应的内部流程                             | 适合场景                                                                 |
| ---------------------------------------------------------------------------- | ----------------------------------- | -------------------------------------------------------------------- |
| “根据 xxx 文档补全开发计划”“把需求拆成实施阶段”“生成迁移计划”“按依赖顺序拆任务”“整理执行计划”“把这个需求变成可执行计划”“先总纲后细化” | `code-helper-plan-workbench`        | 有需求文档、跨模块改动、迁移重构或需要多人持续推进时                                           |
| “补一份人工验收清单”“整理手工测试步骤”“页面验收只写手工测试”“补全回归测试范围”“真实浏览器链路由用户验收”                    | `code-helper-manual-test-workbench` | 页面、可视化、真实浏览器流程、业务验收或自动化不适合覆盖的测试                                      |
| “完成前做一次收尾检查”“检查这个小节点是否完成”“准备最终回复前检查”“切换下一个任务前检查”“看看还能不能归档”                   | `code-helper-completion-review`     | 完成一个小节点、准备最终回复、准备切换任务或判断是否能归档前                                       |
| “这个功能结束了，先问我是否归档”“把已完成任务移到 archive”“查看当前任务状态”“区分活动任务和已归档任务”                  | `code-helper-document-archive`      | 功能已经完成，需要把 plan/result/status 移到 archive，或需要区分 active、archived、mixed |
| “这个规则以后都这样”“先问我是否沉淀到长期记忆”“把这次稳定规则整理成记忆草案”“整理 AGENTS.md”“同步入口文档”              | `code-helper-memory-tuning`         | 本次变更形成了长期协作规则、测试策略、项目结构约定或发布流程约束                                     |
| “你现在是执行子代理”“不要独自在代码库中工作”“按 Agent 协作规范执行”“多 agent 协作推进”“主会话只审阅和汇总”            | `code-helper-agent-collaboration`   | 主会话需要拆分任务、派发执行子代理、审阅结果，或多人/多 agent 交替推进同一项目                          |

也可以把自然表达和命令结合起来说：

> 我已经运行 `npx code-helper plan docs/登录页需求.md 新增登录页`，请用计划相关的 code-helper skill 补全执行计划，再按当前节点推进。

如果团队已经熟悉内部名称，再显式补充：

> 请使用 `code-helper-plan-workbench` 读取需求和现有代码，把这个计划补成可执行版本。

不推荐只说：

- “帮我改一下。”
- “弄个计划。”
- “测试一下。”
- “收尾。”

这些说法太宽泛，agent 可能需要额外判断意图。不是不能用，但更容易漏掉过程文档、手工测试或完成检查。

## Hooks 和 Git hooks 的区别

这两个名字相似，但用途完全不同。

### Agent hooks

Agent hooks 安装到 Codex 或 Claude Code 的项目级配置里：

- Codex：`.codex/hooks.json`
- Claude Code：`.claude/settings.json`

它们在 agent 生命周期中提醒完成检查，运行的是 code-helper 的完成检查包装脚本。这个 hook 的目标是提醒 agent：结束前看看当前任务是否还有未处理事项。

Agent hooks 不替代 agent 自己判断，也不自动归档、不自动更新长期记忆、不自动提交。

### Git hook

Git hook 安装到：

```text
.git/hooks/pre-commit
```

它只在提交前运行：

```bash
npx @skrupellose/code-helper check
```

它的目标是提交前兜底检查协作文档结构，不负责对话收尾，也不判断当前功能是否该归档。

### 避免误装

`init` 可以按选中的 agent 目标安装 Agent hooks，但不会自动安装 Git hook。Git hook 必须显式执行：

```bash
npx code-helper hooks install git
```

如果你只是想让 agent 结束前提醒完成检查，安装 Agent hooks。只有你希望每次 Git 提交前都跑协作文档检查，才安装 Git hook。

## 示例一：新增登录页

一个自然流程可以是：

1. 准备需求文档：`docs/登录页需求.md`。
2. 打开 `npx code-helper` 菜单，选择“项目准备 / 初始化/刷新项目配置”，新项目按实际使用的 agent 选择 Codex、Claude Code 或 GitHub Copilot。
3. 在菜单里选择“任务推进 / 生成任务计划模板”，拖入 `docs/登录页需求.md`，功能名填写“新增登录页”。
4. 对 agent 说：“根据 `docs/登录页需求.md` 补全完整文档”
5. 对 agent 说：“开始按照文档优先级执行开发任务”。
6. 完成后对 agent 说：“做一次收尾检查，确认是否还有未完成项，并更新实施记录。”
7. 如果功能整体完成，就按照agent提示确认归档操作即可。若还有问题，则继续让ai执行开发任务，然后重复6-7步骤，直到完成
8. 告知agent开始执行下一个任务

## 示例二：Go 或 Java 服务接入

假设这是一个 Go 服务或 Java 服务仓库，团队不想引入 `package.json`，可以这样接入：

1. 打开菜单：

```bash
npx @skrupellose/code-helper@latest
```

2. 在菜单里选择“项目准备 / 初始化/刷新项目配置”，按当前使用的 agent 选择 Codex、Claude Code 或 GitHub Copilot。
3. 在菜单里选择“工具设置 / 管理项目 Skills”，注册当前项目实际使用的 agent 目标。
4. 准备需求文档，例如 `docs/库存同步需求.md`。
5. 在菜单里选择“任务推进 / 生成任务计划模板”，拖入需求文档，功能名填写“库存同步改造”。
6. 对 agent 说：“根据 `docs/库存同步需求.md` 和现有 Go/Java 服务结构补全执行计划，自动验证只运行适合本项目的纯逻辑测试，涉及人工业务验收时只补手工测试文档。”
7. 完成节点后，在菜单里选择“任务推进 / 检查功能完成情况”；功能整体完成并经用户确认后，再选择“项目维护 / 归档已完成任务”。

需要写进 README 或交接说明时，可以补充命令版：

```bash
npx @skrupellose/code-helper@latest init codex
npx @skrupellose/code-helper@latest skills register codex
npx @skrupellose/code-helper@latest plan docs/库存同步需求.md 库存同步改造
npx @skrupellose/code-helper@latest finish 库存同步改造 --check-only
npx @skrupellose/code-helper@latest check
```

如果团队使用 Claude Code 或 GitHub Copilot，把示例中的 `codex` 换成 `claudecode` 或 `githubcopilot`。如果仓库明确同时支持多个 agent，再选择 `all`。

## 常见误区

### 误区一：把 plan 当成完整计划

`plan` 是模板生成器，不是智能计划完成器。完整计划需要 agent 读取需求、代码、规则和已有任务状态后补全。

### 误区二：把 manual-test 当成完整验收文档

`manual-test` 只生成结构。完整步骤必须由 agent 结合上下文补全，页面和真实浏览器链路最终由用户手工执行。

### 误区三：注册了 `.code-helper/skills/` 就以为 agent 能看到

`.code-helper/skills/` 是模板源。需要执行 `skills register`，把 Skills 写到 `.agents/skills/`、`.claude/skills/` 或 `.github/skills/`。

### 误区四：Git hook 和 Agent hook 混用

Agent hook 是对话生命周期提醒，Git hook 是提交前检查。想要完成检查提醒，不要安装 Git hook；想要提交前结构检查，才安装 Git hook。

### 误区五：完成后自动更新长期记忆或自动归档

code-helper 的收尾原则是先判断、再询问。长期记忆和归档都需要用户确认，不能由 agent 自动代办。

## 推荐日常节奏

单 agent 项目的默认节奏是先打开菜单：第一次接入走“项目准备 / 初始化/刷新项目配置”，之后到“工具设置 / 管理项目 Skills”注册当前 agent。

如果要写进 Node 项目的 README 或 npm 脚本，对应命令是：

```bash
npm i -D @skrupellose/code-helper
npx code-helper init codex
npx code-helper skills register codex
```

Go、Java 等非 Node 项目可以把命令写成：

```bash
npx @skrupellose/code-helper@latest init codex
npx @skrupellose/code-helper@latest skills register codex
```

每个较完整的功能：

先在菜单里选择“任务推进 / 生成任务计划模板”，需要人工验收时再选择“任务推进 / 生成手工测试模板”，完成前选择“任务推进 / 检查功能完成情况”，用户确认后再选择“项目维护 / 归档已完成任务”。

对应命令可以写成：

```bash
npx code-helper plan docs/需求.md 中文功能名
# 让 agent 补全计划并执行
npx code-helper manual-test 中文功能名
# 仅在需要人工验收时生成并补全
npx code-helper finish 中文功能名 --check-only
# 用户确认后
npx code-helper archive 中文功能名
```

阶段结束或提交前：

```bash
npx code-helper check
git diff --check
```

升级 code-helper：

```bash
npm i -D @skrupellose/code-helper@latest
npx code-helper update
```

保持这个节奏，code-helper 的价值会比较稳定：规则在入口和专题文档里，任务状态在 status-doc 里，历史和验证在 result-doc 里，完成任务进入 archive，agent 通过 Skills 知道什么时候该读什么、写什么、问什么。
