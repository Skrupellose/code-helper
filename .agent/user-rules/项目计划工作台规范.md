# 项目计划工作台规范

## 功能描述

把完整需求文档拆成可推进、可记录阻塞、可恢复上下文、可支持回归的执行工作台。目标不是只写说明文档，而是生成 agent 和开发者可以持续使用的计划、状态和结果记录体系。

## 调用时机

- 用户提供完整需求文档并要求拆分计划
- 项目迁移、框架升级、多阶段功能建设或技术债清理
- 页面、组件、接口、状态层存在明显依赖关系的任务
- 需要长期推进、多人协作或中途暂停再恢复的任务

## 调用入口文件

- `AGENTS.md`

## 规则

1. 计划文档写入 `.agent/plan-doc/<feature>.md`。
2. 执行记录写入 `.agent/result-doc/<feature>/implementation.md`。
3. 当前状态写入 `.agent/status-doc/<feature>-status.md`。
4. 页面测试文档写入 `.agent/result-doc/<feature>/manual-test.md`。
5. 最终计划必须同时具备总纲、分层顺序、组件计划、页面计划、验收标准、状态跟踪、阻塞回归入口和当前推进建议。
6. 生成顺序必须是：目标与约束、P0/P1 总纲、依赖顺序、目录或模块策略、基础阶段计划、组件计划、页面计划、执行工作台。
7. 不要一开始直接写细表；先给总纲和阶段边界，再逐步细化。
8. 必须按依赖闭环重排顺序，不按页面直觉排序。
9. 推荐依赖顺序：基础目录、类型定义、API / 请求层、状态与守卫、应用骨架、i18n 最小闭环、通用组件与布局层、组件组、页面、回归与横向升级。
10. 页面不是第一迁移单元；进入页面计划前必须先生成页面到组件依赖总表、组件分组、组件优先级、单组件 checklist 和组件组真实执行顺序。
11. 页面逐页执行表必须包含前置依赖、执行任务、UI 验收、功能验收和完成定义。
12. 所有总览表和明细表统一使用“子任务 / 已完成 / 未完成 / 备注”四列。
13. 备注第一句必须写 `状态：未开始`、`状态：进行中`、`状态：部分完成`、`状态：被阻塞` 或 `状态：已完成`。
14. 备注必须说明阻塞点、依赖后续流程、回归入口，以及是否允许跳过继续推进。
15. 文档前部必须包含“当前推进建议”，明确现在先做什么、下一步接什么、第一个组件组何时开始、第一个页面何时开始、当前默认执行主线是什么。
16. 页面相关验收必须拆成 UI 验收和功能验收，并生成严格手工测试文档；工具自身只执行纯逻辑测试。
17. 文档只有满足可直接推进、可记录阻塞、可恢复上下文、可支持回归时，才算成型。
