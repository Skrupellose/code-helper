# 项目计划管理规范

## 功能描述

把完整需求文档拆成可推进、可记录阻塞、可恢复上下文、便于后续复查的执行计划。目标不是只写说明文档，而是生成 agent 和开发者可以持续使用的计划、状态和结果记录体系。

## 调用时机

- 用户提供完整需求文档并要求拆分计划
- 项目迁移、框架升级、多阶段功能建设、平台能力建设、数据任务、工具链建设或技术债清理
- 多模块、多系统、多角色、多数据流或跨端交付存在明显依赖关系的任务
- 需要长期推进、多人协作或中途暂停再恢复的任务

## 调用入口文件

- `AGENTS.md`
## 规则

1. 所有计划、结果、状态和测试文档必须使用中文命名与中文总结。
2. 计划文档写入 `code-helper-docs/plan-doc/<中文功能名>.md`。
3. 执行记录写入 `code-helper-docs/result-doc/<中文功能名>/实施记录.md`。
4. 当前状态写入 `code-helper-docs/status-doc/<中文功能名>-状态.md`。
5. 需要人工验收时，手工测试或验收文档写入 `code-helper-docs/result-doc/<中文功能名>/手工测试.md`。
6. 当用户只提供英文功能名时，优先从需求文档的中文一级标题推断中文功能名；仍无法推断时使用中文兜底名。
7. 最终计划必须同时具备总纲、分层顺序、模块或能力拆分、依赖与集成计划、验收标准、状态记录、阻塞点、后续检查点和下一步建议。
8. 生成顺序必须是：目标与约束、P0/P1 总纲、依赖顺序、目录或模块策略、基础阶段计划、核心实现计划、集成验收计划、执行计划。
9. 不要一开始直接写细表；先给总纲和阶段边界，再逐步细化。
10. 必须按依赖链路重排顺序，不按需求文档顺序、界面直觉或个人偏好排序。
11. 推荐依赖顺序：目标和范围、现状和影响面、基础结构、数据模型或接口契约、核心业务规则、关键模块或服务、集成链路、验证方案、发布和交接整理。
12. 不预设任务一定是前端页面或组件；应按实际需求拆成功能模块、领域能力、接口、命令、任务、作业、页面、组件、数据流或服务单元。
13. 集成和验收计划必须包含前置依赖、执行任务、验证方式、验收标准、完成定义和后续检查点。
14. 所有总览表和明细表统一使用“子任务 / 已完成 / 未完成 / 备注”四列。
15. 备注第一句必须写 `状态：未开始`、`状态：进行中`、`状态：部分完成`、`状态：被阻塞` 或 `状态：已完成`。
16. 备注必须说明阻塞点、依赖后续流程、后续检查点，以及是否允许跳过继续推进。
17. 文档前部必须包含“下一步建议”，明确现在先做什么、下一步接什么、第一个核心模块或能力何时开始、第一个集成或验收环节何时开始、当前默认执行主线是什么。
18. 验证计划必须按任务类型选择；纯逻辑、数据转换、接口契约、CLI 命令、权限规则和后端服务优先规划可自动执行的单元测试或集成测试。
19. 页面、可视化、组件交互、视觉和真实浏览器链路只生成严格手工测试文档；工具自身只执行纯逻辑测试。
20. 文档只有满足可直接推进、可记录阻塞、可恢复上下文、便于后续复查时，才算完整。
21. 计划文档和 plan skill 的中文表述必须贴近中文项目管理语境，避免使用“工作台”“驾驶舱”“收口”“当前推进建议”等生硬词；优先使用“计划文档”“状态记录”“完成整理”“下一步建议”等自然表达。
