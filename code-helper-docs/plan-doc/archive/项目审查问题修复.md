# 项目审查问题修复

## 下一步建议

当前默认主线是先修复会造成误操作或误判的 CLI 行为，再调整计划文档生成边界，最后补齐检查、发布和测试覆盖。完成后运行 `npm test`、`npm run check` 和 `npm pack --dry-run` 验证。

## 目标与约束

- 修复产品开发者和用户视角 review 中确认的功能缺陷与体验问题。
- 保持 Git hook、Agent hook、Skills 注册和 init 目标选择边界清晰。
- 计划文档默认不强行生成页面手工测试文档。
- 已归档任务必须被识别为已结束，mixed 冲突必须有明确恢复入口。
- 检查命令默认只读，发布包必须有自动构建保护。
- 不提交、不推送、不发布 npm。

## 验收标准

- `hooks install` 不带目标不会安装任何 hook。
- `createPlanWorkbench` 不再默认创建 `手工测试.md`。
- archived 任务在完成检查中显示为已归档结束状态。
- mixed 归档冲突默认给出清晰错误，显式 `--resolve-mixed` 可恢复。
- `check` 默认不写 `.code-helper/checks/latest.json`。
- `npm pack --dry-run` 自动触发构建。
- 自动化验证全部通过。
